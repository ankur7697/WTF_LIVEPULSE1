# Database

## Schema

The database is initialized from the SQL files in [`backend/src/db/migrations/`](../backend/src/db/migrations).

### Tables

- `gyms`
  - Gym metadata, location, capacity, and operating hours
- `members`
  - Member profile, plan type, status, and last check-in time
- `checkins`
  - In/out records with a generated `duration_min` column
- `payments`
  - Payment events with amount, plan type, and timestamps
- `anomalies`
  - Warning and critical alerts, plus resolved and dismissed flags

## Indexes

- `idx_members_churn_risk`
  - Partial index on `members(last_checkin_at)` where status is active
- `idx_members_gym_id`
  - Supports gym-scoped member lookups
- `idx_checkins_time_brin`
  - BRIN index on `checkins(checked_in)` for time-range scans
- `idx_checkins_live_occupancy`
  - Partial index for open check-ins used by the live occupancy query
- `idx_checkins_member`
  - Member history lookup by check-in time
- `idx_payments_gym_date`
  - Gym-scoped revenue rollups
- `idx_payments_date`
  - Cross-gym revenue comparisons by date range
- `idx_anomalies_active`
  - Partial index for unresolved anomalies

## Materialized View

`gym_hourly_stats` stores recent hourly check-in counts for each gym, day of week, and hour of day.

It exists because the heatmap query groups recent check-ins over a window and should not do that work on every dashboard refresh.

## Seed Data

The seeded dataset is generated by [`backend/src/db/seeds/generateSeedData.js`](../backend/src/db/seeds/generateSeedData.js).

Default seed shape:

- 10 gyms
- 5,000 members
- 90 days of activity
- recent check-ins, checkouts, payments, and anomalies

## Why This Shape Works

- The schema keeps the operational entities separate so the dashboard can query each stream directly.
- The partial indexes keep the hottest live queries small.
- The materialized view keeps the analytics panel fast without losing the freshness required for the demo.
